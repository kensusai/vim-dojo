# モデル間の性能差を埋める(憑依ループ)

上位モデル(例: Claude Fable)の「自律的に進める力」の多くは、能力ではなく**行動規律**(検証する・正直に報告する・止まらない)です。規律は文書に固定できるので、差分を言語化して CLAUDE.md / skills に書き足せば、下位モデル(例: Opus / Sonnet)でも成果物の品質を大きく引き上げられます。

## いつ使うか

- 日常タスクを安いモデル・速いモデルに任せたいが、品質が落ちるとき。
- 特定タスク(記事作成・定型実装・レビュー等)を量産フェーズに移すとき。

## 手順(1ループ)

1. **同じお題を両モデルに実行させる。** 上位モデルの出力をお手本として保存する。
2. **審査観点を先に3つ程度固定する。** 例: ①成果物として通用するか ②提出前に自己確認したか ③できていない確認を正直に申告したか。観点を審査員任せにすると毎回違う指摘が来て発散する。
3. **独立した審査員に比較させる。** お手本を10点に固定し、もう一方を観点ごとに採点 + **差分を文章で言語化**させる。審査員は必ず作業セッションと別系統にする:
   ```bash
   claude -p "以下の2つの成果物を比較してください。Aをお手本として10点に固定し、Bを観点ごとに採点し、差分を具体的に言語化してください。観点: (1)... (2)... (3)... --- A: <お手本> --- B: <対象>"
   ```
   別ベンダーのエージェント(Codex 等)にも同じ依頼をすると、審査員自身の癖を相殺できる。
4. **言語化された差分をルールに変換して書き足す。** 振り分けが重要:
   - どのタスクにも当てはまる規律(検証しない・正直に言わない等)→ **CLAUDE.md / AGENTS.md**(`self-verification` プリセットの導入も検討)
   - そのタスク固有の合格条件 → **タスク特化の skill**(`skills/<task>-self-review/SKILL.md` 等)
   - 可能な限り**機械的に確認できる形**に落とす(「文字数を実測して報告」「禁止表現との突き合わせ」「テスト実行結果を添付」)。曖昧な訓示(「丁寧に書く」)は効かない。
5. **再実行して再評価する。** 点数の推移を記録し、上がらなくなったら止める(だいたい2〜3周で収束する)。

## ルール化のコツ

- 1回の指摘 = 1ルール。まとめて抽象化しない(「注意深く」に丸めると効果が消える)。
- ルールを足すたびに「これは他のタスクでも正しいか?」を問う。No ならタスク特化 skill へ。
- CLAUDE.md の汎用ルールは10個程度まで。肥大化すると全ルールの遵守率が下がる。

## 限界(正直に)

- 移せるのは規律まで。判断の質(設計の筋・根本原因を見抜く深さ)は移らないので、**判断が重い工程(プラン作成・設計・ADR)は上位モデル、実行は手順書に従う下位モデル**という分業を併用する(`docs/plans/` + `/plan-next` がその仕組み)。
- 審査員も間違える。点数そのものより「言語化された差分」を成果物として扱うこと。
