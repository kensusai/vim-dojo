# フェーズ 13: 監視設計

## 目的

「動くものを作る」から「運用できるものにする」への橋渡しとして、監視・運用の設計を `docs/monitoring.md` に整理する。障害時に何を見て、誰が、どう動くかを事前に決めておく。

## 進め方

1. `docs/project-overview.md` の非機能要件(可用性・性能)を確認し、監視すべき水準を決める。
2. `docs/monitoring.md` を作成し、以下を記載する。
3. 実装済みのログ・メトリクスと突き合わせ、不足している計測ポイントをタスク化する。

## docs/monitoring.md に含める内容

- **ログ設計** — レベルの使い分け、構造化フォーマット、相関ID、出力禁止情報(秘密情報・個人情報)
- **メトリクス** — 計測する指標(リクエスト数・レイテンシ・エラー率・ビジネス指標)と計測点
- **エラー通知** — エラートラッキングの送信先、通知の重大度分け
- **パフォーマンス監視** — 監視対象のエンドポイント/処理と目標値(非機能要件との対応)
- **死活監視** — ヘルスチェックエンドポイントと監視間隔
- **監査ログ** — 記録する操作、保持期間、閲覧権限
- **バックアップ** — 対象・頻度・保持期間・リストア手順(リストアの検証方法まで)
- **アラート条件** — 閾値・通知先・営業時間内外の扱い。「アラートが鳴ったら必ず行動が伴う」条件だけ設定する
- **障害対応手順** — 検知 → 一次対応 → エスカレーション → 復旧確認 → 振り返り、の流れと役割

## ルール

- 新機能の実装時は、その機能の失敗をどう検知するか(ログ・メトリクス・アラート)を実装に含める。
- 対応不要なアラートを放置しない。閾値を直すか削除する。
