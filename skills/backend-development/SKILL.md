# フェーズ 9: バックエンド開発

## 目的

バックエンドの実装ルールを `docs/backend.md` に定義し、すべてのバックエンド実装をそのルールに沿って行う。ルールが未定義の領域に触れる場合は、まずルールを提案してから実装する。

## 進め方

1. `docs/architecture.md`(レイヤー・依存方向)と `docs/database.md` を確認する。
2. `docs/backend.md` が無ければ以下の内容で作成し、あれば実装前に該当項目を確認する。
3. 実装は小さな単位で進め、各単位でテストと Lint を通す。

## docs/backend.md に含める内容

- **採用アーキテクチャ** — `docs/architecture.md` のバックエンド部分への適用(レイヤーとディレクトリの対応)
- **ディレクトリ構成** — どこに何を置くか、新規ファイルの置き場所の決め方
- **API設計** — URL/命名規則、リクエスト/レスポンス形式、バージョニング、ページネーション
- **エラーハンドリング** — エラーの分類(業務エラー/システムエラー)、エラーレスポンス形式、リトライ可否の伝え方
- **バリデーション** — どの層で何を検証するか(境界で構造を検証し、内側は型/ドメインで守る)
- **認証・認可** — 認証方式、認可チェックの置き場所(すべての操作でサーバー側チェック必須)
- **バックグラウンドジョブ** — キュー/スケジューラの使い方、冪等性・リトライの方針
- **ログ** — ログレベルの使い分け、構造化ログ、出力してはいけない情報(秘密情報・個人情報)
- **テスト方針** — 層ごとのテスト戦略、外部依存の扱い

## ルール

- 新しいエンドポイント・操作には、認可チェックと少なくとも1つの異常系テストを必ず含める。
- エラーを握りつぶさない。分類して処理するか、明示的に上位へ伝える。
- ルールが未定義の判断に出会ったら、`docs/backend.md` への追記案とセットで実装を提案する。

## クリーンアーキテクチャ: バックエンドの依存方向

- ユースケース・ドメイン層に DB / HTTP / フレームワークの import が入っていないかを、実装のたびに確認する。
- 新しい外部連携は、まず内側の層にポート(インターフェース)を定義し、インフラ層に実装を置く。

## パフォーマンス: バックエンドの必須要件

- エンドポイントを設計するとき、**応答時間の目標**(例: p95 500ms)を決めて `docs/backend.md` に記録する。「速いといいな」で作らない。
- 一覧を返す API は最初から**ページング(またはカーソル)必須**。LIMIT なしの全件クエリを書かない。
- 関連データを含む一覧は実装時に**発行クエリ本数を確認**する(1 + N になっていないか)。ORM は eager loading / JOIN / バッチ取得を明示する。
- WHERE / ORDER BY / JOIN に使う列は、migration でインデックスをセットにする。重いクエリは `EXPLAIN` の結果を PR に貼る。
- 依存関係のない外部呼び出し・クエリは並列化する。ループ内の I/O は書かない。
- 主要エンドポイントには、応答時間かクエリ本数をアサートする**性能回帰テスト**を用意する(閾値は目標値から)。

## 型安全: API境界のパース

- すべてのエンドポイントで、リクエストを境界で型付き構造にパースしてから処理する。ハンドラの内側で生のリクエストオブジェクトを触らない。
- レスポンスの形も型で定義し、API設計(`docs/backend.md`)のスキーマと一致させる。
