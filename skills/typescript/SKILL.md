# TypeScript 開発ガイド

このプロジェクトで TypeScript のコードを書く・変更するときの作法です。提案ではなく、すべての変更に適用されるルールです。原則から外れる場合は、その理由を PR・作業ログに明記します。

## 型の厳格さ(境界を守る)

- `any` を新規に書かない。型が不明な値は `unknown` で受けて絞り込む。
- 外部データ(API レスポンス・`JSON.parse`・環境変数・フォーム入力)は**境界でパース・検証してから**型を付ける。スキーマバリデーション(zod / valibot 等)が導入済みならそれを使う。`as T` で信じ込まない。
- 型アサーション(`as`)より型ガード(`typeof` / `in` / `instanceof` / ユーザー定義型ガード)。`as` の連鎖が必要になったら型設計を疑う。
- `@ts-ignore` は使わない。どうしても抑止が必要なら `@ts-expect-error` + 理由コメント。
- tsconfig の strict 系フラグを緩めない。`strict: false` への変更や `exclude` の追加はレビューで必ず理由を問う。

## 型設計

- **不正な状態を表現できない型にする**: boolean フラグの組み合わせより判別可能ユニオン。例: `{ status: 'loading' } | { status: 'success'; data: T } | { status: 'error'; error: E }` なら「success なのに data が無い」がコンパイル時に消える。
- エクスポートする関数は戻り値型を明示する。推論任せだと、実装変更で公開APIの型が黙って変わる。
- 派生型はユーティリティ型(`Pick` / `Omit` / `Partial` / `ReturnType`)で作り、手でコピーしない(元の変更に追従しなくなる)。
- 文字列の集合は `enum` より union 型(`'a' | 'b'`)か `as const`(既存の慣例があればそれに従う)。
- ジェネリクスは実際に複数の型で使うときだけ導入する。1用途なら具体型で書く。

## null / undefined

- non-null assertion(`!`)は、直前のコードから非 null が保証できるときだけ。迷ったら早期 return か型ガード。
- `??` と `||` を使い分ける(`||` は `0`・`''`・`false` も潰す)。
- optional(`?:`)を「あとで埋める」の意味で使わない。生成時に値を決められるなら必須プロパティにする。

## 実装作法

- 型エラーは、エラーが出た場所ではなく**原因**を直す。アサーションで黙らせるのは最後の手段(理由コメント付き)。
- データ変換は純粋関数 + 明示的な入出力型に切り出す。テストも型チェックも効きやすくなる。
- 型定義の別ファイル(`types.ts`)分離は、複数ファイルから参照されるときだけ。1ファイル内でしか使わない型はそのファイルに置く。

## バージョン追従

- バージョンは記憶に頼らず実測する: `npx tsc --version`、最新は `npm view typescript version`。
- TypeScript の更新で非推奨(deprecation)や新しい strict 系フラグが増えたら、同じプラン(フェーズ)で対応する。ビルドツール(tsup / vite / esbuild 等)の型まわりの挙動変化も一緒に確認する。

## レビューチェックリスト

**機械で取れる指摘はツールに任せる**: まず型チェック(`npm run typecheck`)と Lint(`npm run lint`)を通してから、以下を目視する。

1. **抑止の新規追加** — 新しい `any` / `as` / `!` / `@ts-expect-error` に理由があるか。
2. **境界のパース漏れ** — fetch / `JSON.parse` / 環境変数の結果を `as T` で信じていないか。
3. **不正状態が表現できる型** — boolean フラグの組み合わせを判別可能ユニオンにできないか。
4. **公開関数の戻り値型**が明示されているか。
5. **`||` と `??` の取り違え** — `0` や空文字が正当な値の経路で `||` を使っていないか。
6. **型とランタイムの乖離** — 型上は non-null なのに実データでは null が来る経路がないか。
7. **tsconfig の緩め変更** — strict 系のオフ・`exclude` 追加が変更に紛れていないか。

## ベストプラクティスの遵守

このスキルに書かれていない事項は、TypeScript の**公式ドキュメントが推奨する現行のベストプラクティス**を既定とする:

- 非推奨(deprecated)API・レガシーパターンを新規コードに書かない。既存コードで見つけたら、黙って踏襲せず移行を提案する。
- プロジェクトが**使っているバージョン**の作法で書く: 未導入バージョンの新 API を使わない。逆に、現行バージョンで推奨の書き方があるのに古いイディオムを選ばない。
- 記憶が曖昧な API・設定・オプションは推測で書かず、公式ドキュメントで確認する(ドキュメント参照ツールが使える環境ならそれを使う)。確認できなかった場合はその旨を明記する。

## 型安全重視プリセット: TypeScript の追加ルール

- `noUncheckedIndexedAccess` を有効にする(配列・`Record` の添字アクセスが `| undefined` になり、境界チェックが強制される)。未導入なら導入を提案する。
- 取り違えやすいプリミティブ(各種 ID・金額など)にはブランド型を検討する: `type UserId = string & { __brand: 'UserId' }`。
- オブジェクトリテラルの型適合は `satisfies` で検証する(`as` と違って余剰プロパティも過不足も検出でき、推論結果も狭いまま保てる)。
- switch の網羅性は `never` でチェックする: `default: { const _exhaustive: never = value; ... }` — ユニオンにケースを足したときコンパイルエラーで気づける。
- スキーマライブラリ(zod 等)を使う場合、型はスキーマから導出する(`z.infer<typeof schema>`)。型とバリデーションを二重管理しない。
