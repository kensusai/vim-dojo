# リファクタリング

## いつ使うか

コードは動くが変更しづらいとき: 重複、不明瞭な構造、肥大化したモジュール。

## 手順

1. **挙動を固定する。** 変更前に、現在の挙動をどう検証できるかを確立する。カバレッジが薄ければ先にテストを追加する。
2. **小さなステップでリファクタリングする。** 各ステップ後もコードが動く状態を保つ。
3. **ステップごとに検証する。**
4. **合意したスコープで止める。** 気づいた改善候補は、変更を広げる代わりに列挙して報告する。

## ルール

- 観測可能な挙動を変えない。途中でバグを見つけたら報告する — 修正は別の変更として行う。
- 公開APIやシグネチャの変更は、明示的に告知せずに行わない。
- 差分をレビュー可能な大きさに保つ。大きなリファクタリングは小さな連続に分割する。

## クリーンアーキテクチャ: 境界の修復

リファクタリング中は境界違反を積極的に探して修正する:

- ドメイン/ユースケースがフレームワーク・ORM・HTTP型を import している → ポートを導入し、実装を外側へ移す。
- ビジネスロジックがコントローラ・ハンドラ・UIコンポーネントに存在する → ユースケースへ抽出する。
- ORMエンティティやリクエストオブジェクトがレイヤーを越えて漏れている → 境界でプレーンな DTO に置き換える。
- レイヤー間の循環参照・外向き import → インターフェースで依存を逆転する。

## パフォーマンスを意識したリファクタリング

- 既知のホットパスをリファクタリングするときは前後で計測する。計測済みの性能を黙って見た目のために犠牲にしない。
- 最適化は局所化し、文書化する。計測で正当化できなくなった最適化は単純化して取り除く。

## 継続的リファクタリングの規律

- スイートをグリーンに保てる最小のステップで進め、コミットできる安定点で区切る。
- その場しのぎの書き直しより、既知の可逆なリファクタリング(関数の抽出・インライン化・リネーム・移動)を優先する。
- リファクタリングが合意したスコープを超えて膨らんだら、拡大せずに立ち止まって報告する。
