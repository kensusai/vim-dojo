# フェーズ 8: CI構築

## 目的

実装が本格化する前に「すべての変更が自動で検証される」状態を作る。CI の構成と方針を `docs/ci.md` に整理し、パイプライン自体を構築する。

## 進め方

1. CI基盤を確定し、`docs/tech-stack.md` に記録する。リポジトリのホストに合わせるのが基本(GitHub → GitHub Actions、GitLab → GitLab CI)。既に CI がある場合は現状を棚卸しし、不足を埋める形で進める。
2. `docs/ci.md` を作成する。
3. パイプラインを実装する(例: `.github/workflows/ci.yml`、`.gitlab-ci.yml`)。実行するコマンドはプロジェクトの実コマンドをそのまま使う。
4. 検証: 設定ファイルの構文チェックを行い、push / PR でパイプラインが実際に走ってグリーンになることを確認する。

## docs/ci.md に含める内容

- **CI基盤の選定と理由** — どの CI か、なぜか
- **パイプライン構成** — ステージ(lint → 型チェック → テスト → ビルド)と実行トリガー(push / PR / タグ)
- **実行環境** — ランナー、言語バージョンのマトリクス(サポート範囲と揃える)
- **キャッシュ戦略** — 依存キャッシュのキーと無効化条件
- **シークレット管理** — CI変数に平文を置かない、OIDC 等の連携方式、ログへの漏えい防止(マスク)
- **ブランチ保護** — 必須チェック、レビュー要件
- **デプロイトリガー** — CD への接続点(どのブランチ / タグで何が起きるか)
- **失敗時の運用** — フレーキーテストの扱い、再実行の方針

## ルール

- CI で実行するコマンドはローカルと同一にする(package.json / タスクランナー経由)。CI 専用の隠しロジックを作らない。
- シークレットをワークフローファイル・ログ・リポジトリに書かない。
- パイプラインの変更もコードと同じくレビュー対象とする。
- 新しいチェックを追加するときは、まずローカルで通ることを確認してから CI に載せる。
