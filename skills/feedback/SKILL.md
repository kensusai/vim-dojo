# agent-ready へのフィードバック

このプロジェクトの CLAUDE.md / AGENTS.md / `skills/` は agent-ready が生成しています。従って作業する中で得た気づきは本体の改善材料になるので、`docs/agent-ready-feedback.md` に記録します。記録は作業のついでに行い、確認を求める必要はありません。

## いつ書くか

- **bug** — 手順書・ルールの記述が誤っている(従うと失敗する・実態と合わない)
- **unclear** — 曖昧で解釈に迷った
- **gap** — 必要な情報・手順が欠けている(このプロジェクト固有ではなく、他のプロジェクトでも使えそうなもの)
- **win** — 特に効いたルール(強化・横展開すべき)
- **new-skill** — スキルが存在しない技術・作業に出会った(後述)

## エントリ形式(厳守 — 機械的に回収される)

```markdown
## <YYYY-MM-DD> | <bug|unclear|gap|win|new-skill> | <対象: skills/python 等、new-skill は提案名>

- 状況: <何をしていて何が起きたか(1〜2行)>
- 提案: <どう直すべきか・何を足すべきか(1〜2行)>
- ステータス: 未回収
```

- 1気づき = 1エントリ。まとめて書かない。
- `ステータス: 未回収` のまま追記する(回収側が「反映済み」「見送り」に書き換える)。既存エントリは編集しない。
- ファイルが無ければこの見出しで作成する: `# agent-ready フィードバック`

## スキルが無い技術に出会ったら(new-skill)

1. **このプロジェクト用のローカルスキルを起案してよい**: `skills/<tech>/SKILL.md` に、そのプロジェクトで守るべき作法を書く。構成は既存のスタックスキルに合わせる(作法 → 使う道具 → テスト → レビューチェックリスト。機械で確認できる指摘はツールに任せる原則も同じ)。
2. あわせて `new-skill` エントリを記録する。提案には「対象技術 / 検出方法(依存パッケージ名・設定ファイル名)/ 起案したローカルスキルのパス」を含める。
3. 回収時に汎用化の価値があると判断されれば、agent-ready 本体の `stacks/` に昇格され、他のプロジェクトにも配布される。

## 書かないもの

- このプロジェクト固有の事情(それは `docs/` に直接書く)
- プロダクトコード自体のバグ(それは通常のバグ修正として扱う)
