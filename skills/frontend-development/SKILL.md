# フェーズ 10: フロントエンド開発

## 目的

フロントエンドの実装ルールを `docs/frontend.md` に定義し、画面実装をデザイン(Figma等)と一貫した形で進める。

## 進め方

1. デザインソース(Figma等)がある場合は、実装前に該当画面のデザインを確認し、デザインとの差分を発生させない。デザインが無い画面は、`skills/design-mockup/SKILL.md` でHTMLモックを作成 → Figma(html.to.design)に取り込んで確定してから実装する(既存画面のパターン踏襲でも可)。
2. `docs/frontend.md` が無ければ以下の内容で作成し、あれば実装前に該当項目を確認する。
3. 新しい画面は「画面遷移図への追加 → コンポーネント分割 → 実装 → a11y チェック」の順で進める。

## docs/frontend.md に含める内容

- **画面設計** — デザインソース(FigmaのURL等)との対応表、デザイントークンの扱い
- **画面遷移図** — 主要フローの遷移(Mermaid)
- **共通コンポーネント** — 一覧と使い分け。新規コンポーネントを作る前に既存を確認するルール
- **state管理方針** — サーバー状態/UI状態/フォーム状態をそれぞれ何で管理するか
- **ディレクトリ構成** — ページ・コンポーネント・フックの置き場所
- **UI実装ルール** — スタイリング手法、命名、既存パターンの踏襲
- **フォーム設計** — バリデーションのタイミング、エラー表示位置、送信中の状態
- **エラーメッセージ** — 文言のトーン、ユーザーが次に取れる行動を示すこと
- **ローディング表示** — スケルトン/スピナーの使い分け、レイアウトシフトの回避
- **アクセシビリティ** — キーボード操作・セマンティックHTML・コントラスト(WCAG AA)・フォーカス管理
- **レスポンシブ対応** — ブレークポイントと確認すべき画面幅

## ルール

- 新規コンポーネントを作る前に、共通コンポーネントで実現できないか必ず確認する。
- 画面の完了条件に「キーボードのみで操作できる」「エラー・ローディング・空状態が実装されている」を含める。
- デザインと実装が食い違う場合は、無断でどちらかに寄せず差分を報告する。

## クリーンアーキテクチャ: フロントエンドの依存方向

- UIコンポーネントにビジネスロジックを書かない。ロジックはユースケース/サービス層に置き、コンポーネントは表示と入力の変換に徹する。
- API レスポンスの型を UI 全体に引き回さず、境界でアプリ内部の型に変換する。

## パフォーマンス: フロントエンドの必須要件

- 画面の性能目標は **Core Web Vitals**(LCP 2.5s / INP 200ms / CLS 0.1)を基準にし、主要画面で Lighthouse を実行して確認する。
- 画像は**実寸に合わせたサイズ + 遅延読み込み**で配信し、`width` / `height`(または `aspect-ratio`)を必ず指定してレイアウトシフトを防ぐ。
- 数百行を超えうるリストは**仮想化**する。全件 DOM 描画のまま「あとで直す」にしない。
- 再レンダーを疑うときはフレームワークのプロファイラ(React DevTools Profiler 等)で**計測してから**直す: 毎レンダーで参照が変わる props、粒度の粗い state、コンテキストの過剰購読が定番。
- 重い依存を追加するときはバンドルサイズへの影響を確認する(bundle analyzer)。画面単位の code splitting / 動的 import を検討する。
- データ取得はキャッシュ・再検証を持つ層(TanStack Query / SWR / フレームワークの機構)に任せ、同じデータの重複フェッチを避ける。

## 型安全: フロントエンドの状態モデリング

- 画面状態は「loading / success / error」等の判別可能なユニオンでモデル化し、boolean の組み合わせ(`isLoading && !error && data` など)で表現しない。
- APIレスポンスは受信した境界でパースし、UI内部にはパース済みの型だけを流す。
