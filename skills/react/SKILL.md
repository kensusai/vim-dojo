# React 開発ガイド

このプロジェクトで React のコードを書く・変更するときの作法です。提案ではなく、すべての変更に適用されるルールです。原則から外れる場合は、その理由を PR・作業ログに明記します。

## コンポーネント設計

- 関数コンポーネント + hooks のみ。クラスコンポーネントは書かない。
- 1コンポーネント1責務 — **責務を1文で説明できること**。説明に「と」が増えたら分割のサイン(目安: 150行超・条件分岐のネスト・props 8個以上)。
- 親(ページ)と子の分担を守る: 親はデータ取得・画面全体の状態・遷移・送信などの具体処理、子は表示とユーザー操作の受け口。**データは親から子へ(props)、イベントは子から親へ(`onXxx`)**。
- 子コンポーネントから API・Storage・グローバル状態の更新を直接呼ばない。親から渡されたハンドラを呼ぶ。
- props には TypeScript で型を付ける。`any` にしない。children を受けるなら `ReactNode`。
- props 命名: 値は名詞(`user`, `selectedId`)、子へ渡すイベントは `onXxx`、受け側のハンドラは `handleXxx`、boolean は肯定形(`open`, `disabled`)。
- バリアントだらけの巨大コンポーネント(props フラグで分岐の山)より、小さいコンポーネントの合成を選ぶ。
- 画面の状態は **Loading / Error / Empty / Success の4つ**を最初から設計する。成功パスだけ作って残りを後付けしない。

## state の作法

- まずローカル state(`useState`)。持ち上げ(lift up)は実際に共有が必要になってから。グローバル化(Context / ストア)は複数画面で本当に共有するときだけ。
- **導出できる値を state にしない**: props や既存 state から計算できる値は render 中に計算する(重ければ `useMemo`)。state のコピーを `useEffect` で同期し合う構造は設計ミス。
- 「props ミラー」を書かない: `useEffect(() => setX(propX), [propX])` は state の二重管理。props をそのまま使う。
- サーバーデータを `useEffect` + `useState` で自前フェッチしない。TanStack Query / SWR / フレームワークのデータ取得(Next.js の Server Components・Remix の loader)に任せる。キャッシュ・再検証・レース処理を手書きしない。
- グローバル state・フォームライブラリの導入は既存構成に従う。新規導入は理由とセットで提案する。

## useEffect は最終手段

- 迷ったら**2つの質問**で決める:
  1. **画面の外に触るか?**(API / DOM / Storage / タイマー / 購読)— No なら render 中の純粋計算(重ければ `useMemo`)。
  2. **起点はユーザー操作か?**(クリック・入力・送信)— Yes ならイベントハンドラに書く。
     両方 No(表示・マウントに伴う外部同期)のときだけ `useEffect`。
- 書く場合: 依存配列に嘘をつかない(`exhaustive-deps` 警告を抑止しない)、購読・タイマー・接続を開始したら**必ずクリーンアップ関数で解除する**(解除漏れはメモリリーク・二重購読の直接原因。開発時の Strict Mode は Effect を2回実行するので、クリーンアップが無いと開発中から壊れる)、非同期処理は `AbortController` か ignore フラグでアンマウント後の setState・レースを防ぐ。

## カスタムフックの切り出し

- hooks を1つも使わないロジックをカスタムフックにしない — **純粋関数**として切り出す。どこからでも呼べて、テストも入出力の検証だけで済む。
- 切り出しを検討するサイン: `useEffect` が2つ以上になり責務が絡み始めた / 同型ロジックが3箇所に出た(2箇所なら候補)/「開始 → 進行 → 成功/失敗」の状態遷移が1セットになっている。
- 命名は `use` + 対象 + 役割(`useUsersQuery`, `useWindowSize`)。返す値が3つ以上ならオブジェクトで返す。
- 置き場所: 1画面専用ならその画面のディレクトリにコロケーション。共通ディレクトリへの昇格は**2箇所目が実際に現れてから**(先回りで共通化しない)。

## パフォーマンス

- `memo` / `useMemo` / `useCallback` は**計測してから**。先回りの memo 化は読みにくさに対して見返りがない。
- リストの `key` に配列 index を使わない(並べ替え・挿入で state が化ける)。安定した ID を使う。
- 巨大リストの仮想化・コード分割も、実測で問題になってから。

## アクセシビリティ

- クリックできるものは `<button>`。`<div onClick>` を書かない。ページ遷移は `<a>` / フレームワークの Link。
- フォーム input には label を関連付ける。画像には alt(装飾なら `alt=""`)。
- `nav` / `main` / `section` などの semantic 要素を優先する。

## テスト(Testing Library)

- ユーザー視点でテストする: クエリは `getByRole` を第一候補に、見えるテキストと振る舞いを検証する。
- 実装詳細(state の中身・内部関数の呼び出し回数)をテストしない。リファクタリングで壊れないテストにする。
- 操作は `userEvent`(`fireEvent` より優先)。非同期の出現待ちは `findBy*` / `waitFor`。
- 優先順位: 初期表示 → 入力・バリデーションエラー → 送信の成功/失敗。全網羅より主要動作を確実に。フックから切り出した純粋関数は普通の関数として入出力をテストする。
- タスク完了前に必ずテストを実行する: `npm run test`

## バージョン追従

- 最新版は記憶に頼らず実測する: `npm view react version`(Next.js は `npm view next version`)。
- React / Next.js のメジャー更新には公式 codemod を使う: `npx react-codemod`、Next.js は `npx @next/codemod@latest`。
- 更新で出た非推奨警告は放置せず、同じプラン(フェーズ)で解消する。

## レビューチェックリスト

React コードのレビュー・自己チェックの観点。**機械で取れる指摘はツールに任せる**: まず Lint(`npm run lint`)を実行し、特に `react-hooks/exhaustive-deps` の警告を確認してから以下を目視する。

1. **useEffect の妥当性** — 2つの質問(外部に触るか・ユーザー操作起点か)に照らしてそもそも不要ではないか、依存配列の嘘、クリーンアップ漏れ、非同期レース(アンマウント後の setState)。
2. **state の過剰** — 導出できる値が state になっていないか、props ミラー(`useEffect` で props を state にコピー)、`useEffect` で同期し合う state のペア。
3. **key** — index key・非安定 key(`Math.random()` 等)。
4. **再レンダリング爆発** — Context に巨大な値を丸ごと入れていないか、render 中に生成したオブジェクト・関数を memo 化済みコンポーネントに渡して無効化していないか(指摘は計測とセットで)。
5. **サーバー/クライアント境界(Next.js)** — `'use client'` の付けすぎで木全体がクライアント化していないか、サーバーでできる処理をクライアントでしていないか、シークレットがクライアントバンドルに漏れていないか(`NEXT_PUBLIC_` の誤用)。
6. **表示状態と操作の頑健さ** — Loading / Error / Empty / Success の4状態が考慮されているか、送信中・削除中の二重実行防止があるか。
7. **責務と分担** — コンポーネントの責務を1文で説明できるか、子が API・グローバル状態更新を直接呼んでいないか、hooks を使わないロジックが `useXxx` になっていないか(→純粋関数)。
8. **アクセシビリティ** — div ボタン、label 無し input、キーボードで到達できないインタラクション。
9. **テスト** — 実装詳細のテスト、`getByRole` で書けるのに `getByTestId` に逃げていないか。
10. **簡潔さ** — 利用箇所が1つのカスタムフック、まだ使われていない汎用化(YAGNI)、不要な `memo` の森。

## ベストプラクティスの遵守

このスキルに書かれていない事項は、React(および使用中のメタフレームワーク) の**公式ドキュメントが推奨する現行のベストプラクティス**を既定とする:

- 非推奨(deprecated)API・レガシーパターンを新規コードに書かない。既存コードで見つけたら、黙って踏襲せず移行を提案する。
- プロジェクトが**使っているバージョン**の作法で書く: 未導入バージョンの新 API を使わない。逆に、現行バージョンで推奨の書き方があるのに古いイディオムを選ばない。
- 記憶が曖昧な API・設定・オプションは推測で書かず、公式ドキュメントで確認する(ドキュメント参照ツールが使える環境ならそれを使う)。確認できなかった場合はその旨を明記する。
