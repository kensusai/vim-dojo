# デザインモック作成

## いつ使うか

「こういう画面を作って」という指示から、デザインの叩き台が必要なとき。成果物は自己完結HTMLで、ブラウザで即確認でき、Figma の **html.to.design** プラグインで編集可能なデザインデータに変換できる。確定したモックはそのままUI実装の仕様になる。

## 手順

1. **ヒアリング。** 目的・対象ユーザー・必須要素・トンマナ(参考UI、ブランド色があれば hex で)・対象幅(モバイル 375px / デスクトップ 1440px など)を確認する。曖昧な点は仮定を明示して進める。
2. **モックを作成。** `design/mockups/<画面名>.html` に、下の「HTML作成ルール」に従って書く。複数案を求められたら `<画面名>-a.html` / `-b.html` のように分ける。
3. **確認。** ブラウザで開いて崩れがないことを確認し、ユーザーに見せる(macOS なら `open design/mockups/<画面名>.html`)。フィードバックを反映して確定する。
4. **Figmaへの取り込みを案内。** 確定したら、下の「Figmaへの取り込み手順」を伝える。
5. **実装への橋渡し。** モックの `:root` に定義したデザイントークンとコンポーネント分割案を整理し、`docs/frontend.md` があれば画面設計の節に対応を追記する。実装自体は `skills/add-feature` や frontend-development フェーズで行う(モックが仕様)。

## HTML作成ルール(html.to.design 互換のため必須)

- **1ファイル完結**: CSS は `<style>` タグに書く。外部CSS・CDN・JSフレームワーク禁止。JSに依存した描画をしない(プラグインは静的なDOMとスタイルを変換する)。
- **画像は外部URLを使わない**: プレースホルダは単色/グラデーションの div + ラベルで表現し、必要なら小さな data URI を使う。アイコンはインラインSVG。
- **フォントはシステムフォントスタック**を基本にする(外部フォントはプラグイン側で再現されないことがある)。
- **デザイントークンを `:root` の CSS変数で定義する**(色・余白・角丸・フォントサイズ)。実装時にそのまま流用できる。
- レイアウトは Flexbox / Grid で組み、フレーム幅を明示する(`max-width` 等)。モバイル/デスクトップ両方が必要なら別ファイルに分ける。
- 主要ブロックには意味のある `class` 名を付ける(Figma に取り込んだときのレイヤー名が読みやすくなる)。
- 必要ならエラー・空・ローディングなどの状態バリエーションもモック下部に並べて見せる。
- ダミーデータはそれらしい日本語にする。lorem ipsum は使わない。

## Figmaへの取り込み手順(html.to.design)

1. Figma でファイルを開き、メニュー → Plugins → **html.to.design** を実行する(未導入なら Community から入手)。
2. プラグインの **Paste code**(HTMLコード貼り付け)を選び、モックHTMLの中身を全文貼り付けて Import する。
3. 編集可能なレイヤーとして展開されるので、以降の細かなデザイン調整は Figma 側で行う。
4. Figma 側で確定したデザインは、変更点をモックHTMLか `docs/frontend.md` に反映し、コードとデザインの対応を保つ。

## ルール

- モックは `design/mockups/` 配下に置き、確定前の案もコミットして経緯を残す。
- モック段階では実装コード(React等)に変換しない — 確定後に frontend-development / add-feature で行う。
- デザインと実装が食い違ったら、無断でどちらかに寄せず差分を報告する。
