# フェーズ 4: アーキテクチャ設計

## 目的

プロジェクト全体の設計方針を `docs/architecture.md` に明文化し、重要な設計判断を `docs/adr/` に記録する。以降の実装がこの方針に沿っているかを判定できる状態にする。

## 進め方

1. `docs/project-overview.md` と `docs/domain.md` の要件・制約を確認する。
2. 既存コードがある場合は現状のアーキテクチャを観察し、現状と理想を区別して書く。
3. `docs/architecture.md` を作成する。
4. 大きな設計判断(アーキテクチャスタイル、同期/非同期、モノリス/分割など)は 1 判断 = 1 ADR として `docs/adr/NNNN-<slug>.md` に記録する。
5. 決定した設計ルールを `CLAUDE.md` / `AGENTS.md` のコーディング規約・開発原則に反映する。

## docs/architecture.md に含める内容

- **採用アーキテクチャ** — スタイルと採用理由(例: レイヤード、クリーンアーキテクチャ、モジュラーモノリス)
- **レイヤー構成** — 各レイヤーの責務と置き場所(ディレクトリ対応)
- **依存方向** — 許可される依存と禁止される依存(図または箇条書き)
- **モジュール分割** — 分割単位と境界の根拠(境界づけられたコンテキストとの対応)
- **外部サービス連携** — 連携先・プロトコル・障害時の扱い
- **ADR作成ルール** — いつ書くか・テンプレート・番号の付け方

## ADR テンプレート

```markdown
# NNNN: <タイトル>

- ステータス: 提案 | 採用 | 廃止
- 日付: YYYY-MM-DD

## コンテキスト(なぜ判断が必要か)

## 決定(何をどうするか)

## 検討した代替案(採用しなかった案と理由)

## 影響(この決定で生じるトレードオフ)
```

## ルール

- 依存方向のルールに違反する実装を見つけたら、拡張する前に指摘・修正する。
- アーキテクチャに関わる変更は、実装前に ADR を書いて合意を得る。

## クリーンアーキテクチャ: 設計ドキュメントの必須項目

- `docs/architecture.md` のレイヤー構成は、ドメイン / ユースケース / アダプタ / インフラの4層とディレクトリの対応を必ず定義する。
- 依存方向のルール(内向きのみ)と、違反を検出する方法(レビュー観点・lintルール等)を明記する。
