# 0006: vim エンジンに CodeMirror 6 + @replit/codemirror-vim を採用

- ステータス: 採用
- 日付: 2026-07-11

## コンテキスト(なぜ判断が必要か)

vim エミュレーションエンジンは本アプリの技術的中核で、技術選定の最大論点として残っていた(project-overview.md 未確定事項1)。ADR-0003 で `VimEngine` ポートの向こうに隔離する構造は決定済みのため、ここでは初期実装に使うライブラリを決める。要件: カリキュラム範囲(基本移動 / オペレータ+テキストオブジェクト / 検索・置換 / レジスタ)のカバー、軽量(起動サクサク要件)、キーストロークの計測とバッファ内容の検査が可能なこと。マクロは対象外(要件確定済み)なので再現度の要求は中程度。

## 決定(何をどうするか)

- CodeMirror 6 をエディタ基盤、`@replit/codemirror-vim` を vim エミュレーションとして採用する。
- `src/vim/` で `VimEngine` ポート(core/ports.ts)にラップし、CodeMirror の API・イベント形式を ui / core に一切漏らさない(ADR-0003)。
- ステージ4「複数ファイル/バッファ操作」はエンジンに求めない。バッファ切替・ウィンドウ分割はアプリ側(ui + core)の機能として実装し、エンジンへの要求は常に「単一バッファ」とする。
- 実装初期に「必須コマンド一覧 × 挙動確認」のスパイクを行い、カバレッジ表を `docs/` に残す(レッスン設計の前提になるため)。
- 挙動の基準は Neovim のモダンな標準(domain.md P9)。エミュレータのデフォルトが素の vim 互換で異なる箇所(例: `Y`)は `src/vim/` のラッパーで設定・上書きし、スパイクのカバレッジ表に「Neovim 標準との差異」列を設ける。

## 検討した代替案(採用しなかった案と理由)

- **monaco-vim(Monaco Editor + vim 拡張)**: Monaco は VS Code 由来で高機能だがバンドルが巨大。初回ロード数秒以内・再訪即時という非機能要件に不利。エディタ機能(補完等)はこのアプリでは不要。
- **vim.wasm(本物の Vim を WASM 化)**: 再現度は完璧だが、WASM バイナリのロードが重く、キーストローク単位の計測・バッファ内容のフック・独自 UI(演出)との統合が難しい。学習ゲームには過剰な忠実さ。
- **自前エンジン**: 出題・計測との統合自由度は最高だが、vim の挙動(カウント・リピート・モーションの組み合わせ)の再実装は工数が支配的。ポート化(ADR-0003)により将来の差し替え先として温存する。
- **CodeMirror 5 の vim keymap**: 実績は長いが CM5 系はレガシー。`@replit/codemirror-vim` は同実装の CM6 移植で、こちらが現行。

## 影響(この決定で生じるトレードオフ)

- (+) 軽量・モジュール式で起動サクサク要件と整合。実績のある vim 実装(replit が本番利用)。
- (+) CodeMirror 6 の拡張機構でキー入力のフック・装飾(お題のハイライト等)が実装しやすい。
- (−) 本物の vim と挙動が異なる箇所が必ずある。緩和策: スパイクでカバレッジ表を作り、挙動差異があるコマンドはレッスンから外すか注記する。
- (−) `:g` などの Ex コマンドの対応範囲に制約がありうる。ステージ3の設計時にカバレッジ表と突き合わせる。
- (−) ライブラリのメンテ停滞リスク。緩和策: ポート化により差し替え可能(ADR-0003 の保険がそのまま効く)。
