# 0001: バックエンドレス SPA + ブラウザローカル永続化

- ステータス: 採用
- 日付: 2026-07-11

## コンテキスト(なぜ判断が必要か)

vim-dojo は当面オーナー1名専用の学習ゲームであり、要件(docs/project-overview.md)は「アカウント・サーバー側保存なし」「起動がサクサク」「ランニングコストほぼゼロ」を求めている。アプリの土台となる実行形態を最初に決める必要がある。

## 決定(何をどうするか)

- アプリはブラウザ内で完結する静的配信の SPA とする。サーバー側アプリケーションコード・API・DB を持たない。
- 進捗データ(Progress)はブラウザローカルのストレージに永続化する(具体 API は技術選定フェーズで決定。容量と構造化の観点から IndexedDB を第一候補とする)。
- データ喪失リスクはエクスポート/インポート機能(UC6)で緩和する。
- 静的ホスティング(GitHub Pages 等、インフラ構築フェーズで確定)+ キャッシュにより再訪時の即時起動を実現する。

## 検討した代替案(採用しなかった案と理由)

- **通常の Web アプリ(API サーバー + DB)**: 将来の公開・ランキングには必要だが、初期要件に無く、コスト・複雑さ・起動速度で全て不利。公開が決まった時点で ADR を改訂する。
- **BaaS(Firebase/Supabase 等)**: サーバー管理は不要になるが、認証・ネットワーク往復が入り「ローカル完結・即起動」に反する。
- **デスクトップアプリ(Electron/Tauri)**: オフライン・ローカル保存は自然だが、配布・更新の手間が増え、「ブラウザで開くだけ」の手軽さ(習慣化に直結)を失う。

## 影響(この決定で生じるトレードオフ)

- (+) ランニングコストゼロ、デプロイは静的ファイル配信のみ、起動が速い。
- (+) 進捗データが外部に出ないためプライバシー考慮が不要。
- (−) ブラウザ・端末を変えると進捗が引き継がれない(エクスポート/インポートで手動移行)。
- (−) ブラウザのデータ消去で進捗が消えるリスクが常にある(定期エクスポートの促しで緩和)。
- (−) 将来ランキング等を作る場合はバックエンド追加の再設計が必要(その時点の要件で判断)。
