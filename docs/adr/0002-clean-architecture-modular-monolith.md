# 0002: クリーンアーキテクチャのモジュラーモノリス構成

- ステータス: 廃止(→ [ADR-0005](0005-simple-core-shell-structure.md) に置換)
- 日付: 2026-07-11

## コンテキスト(なぜ判断が必要か)

本プロジェクトはクリーンアーキテクチャを開発原則として採用している(CLAUDE.md)。フロントエンドのみの SPA(ADR-0001)にどう適用するか、またドメイン理解フェーズで特定した5つの境界づけられたコンテキスト(演習・カリキュラム・出題生成・進捗・分析)をどうディレクトリに落とすかを決める必要がある。

## 決定(何をどうするか)

- `src/` 直下をレイヤー第一(`domain/` `usecase/` `adapter/` `infra/`)、その中をコンテキスト第二で分割する。詳細は `docs/architecture.md` のレイヤー構成を正とする。
- 依存は内向きのみ。domain はブラウザ API(`Date.now()` 含む)にも依存せず、時刻・乱数は `Clock` / `RandomSource` ポート経由で注入する。
- domain のコンテキスト間の直接 import を禁止し、コンテキストをまたぐ処理はユースケース層で合成する。
- 依存ルールは lint(dependency-cruiser 等、CI 構築フェーズで確定)で機械的に検査する。

## 検討した代替案(採用しなかった案と理由)

- **コンテキスト第一・レイヤー第二(`src/practice/domain/` 等)**: 機能の凝集は高いが、依存方向の lint 検査がコンテキスト数 × レイヤー数の組み合わせになり複雑化する。トップレベル4ディレクトリならルールが4本で済む。
- **レイヤー分けなしの通常的な SPA 構成(components/ hooks/ utils/)**: 最速で書き始められるが、業務ルール(R1〜R19)が UI に染み出し、vim エンジン差し替えや将来のバックエンド追加が高くつく。開発原則にも反する。
- **コンテキストごとに別パッケージ(monorepo)**: 利用者1名・単一アプリに対して過剰。ビルド設定の複雑さが「サクサク開発」を損なう。

## 影響(この決定で生じるトレードオフ)

- (+) 業務ルールがフレームワーク非依存で単体テストしやすい(受け入れテストの種 = R1〜R19)。
- (+) vim エンジン・UI フレームワーク・ストレージを個別に差し替え可能。
- (−) 小さな機能でも4層を通すボイラープレートが発生する。緩和策: レイヤーを跨ぐ変更のテンプレを skills 化することを検討。
- (−) レイヤー第一構成のため、1機能の変更が複数ディレクトリに散る(トレードオフとして受容)。
