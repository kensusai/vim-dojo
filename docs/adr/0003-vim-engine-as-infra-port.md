# 0003: vim エンジンをポート化しインフラ層に隔離

- ステータス: 採用
- 日付: 2026-07-11

## コンテキスト(なぜ判断が必要か)

vim エミュレーション(モード・モーション・オペレータの実装)はこのアプリの技術的中核だが、「既存ライブラリを使うか自前実装か」は技術選定フェーズの最大論点として未決である(docs/project-overview.md 未確定事項1)。エンジンの選択がドメイン・ユースケースの設計を汚染しない構造を先に決めておく必要がある。

## 決定(何をどうするか)

- vim エミュレーションはドメインではなく技術基盤と位置づけ、`src/vim/` に置く(ディレクトリは ADR-0005 の構成に従う)。
- `core/ports.ts` に `VimEngine` ポート(interface)を定義する。core と ui はこのポートだけに依存する。ポートが提供する最小の能力:
  - バッファの初期化と現在内容の取得(クリア判定 R1 は domain 側で行う)
  - キー入力の送出と、キーストロークイベントの購読(計測 R2 は domain/practice 側で行う)
  - 現在モードの取得(UI 表示用)
- お題(Exercise)・試行(Attempt)・メダル判定はエンジンに一切依存しない純粋なドメインコードとする。
- これにより技術選定で「既存ライブラリ → 不満が出たら自前」という段階的な差し替えが可能になる。

## 検討した代替案(採用しなかった案と理由)

- **エンジンをドメイン層に置く**: 「vim 操作こそドメインでは」という見方もあるが、業務ルール(R1〜R19)はどれも「バッファ内容とキーストローク数」しか参照せず、vim の内部状態には依存しない。エンジンはライブラリ差し替えが起こりうる典型的な「詳細」であり、ドメインに置くと差し替え時の影響が全域に及ぶ。
- **エンジン抽象化なしで直接利用**: 実装は速いが、ライブラリの API・イベント形式が UI とユースケースに直接漏れ、差し替え時に全面書き換えになる。最大の技術リスク(エンジンの再現度不足)に対する保険を失う。

## 影響(この決定で生じるトレードオフ)

- (+) 技術選定の失敗が `infra/vim/` の中に閉じる。エンジン差し替えの影響範囲が明確。
- (+) ドメイン・ユースケースのテストにエンジン不要(フェイクの `VimEngine` で足りる)。
- (−) ポートの抽象を1枚挟む分、エンジンのフル機能(例: ライブラリ固有の拡張)を使いたい時に抽象の改訂が必要。
- (−) キーストローク計測の定義(R2)とエンジンのイベント粒度がズレる場合、アダプタでの変換ルールを慎重に設計する必要がある。
