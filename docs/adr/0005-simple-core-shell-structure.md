# 0005: core/shell の簡易構成(4層ディレクトリの廃止)

- ステータス: 採用(ADR-0002 を置換)
- 日付: 2026-07-11

## コンテキスト(なぜ判断が必要か)

ADR-0002 はクリーンアーキテクチャの4層(domain/usecase/adapter/infra)をトップレベルディレクトリとして採用したが、プロダクトオーナーから「SPA でフル 4層はやりづらい。1つの変更でファイルをあちこち見る必要がある。エッセンスだけ取り入れて、わかりやすいディレクトリ構造でシンプルに実装したい」という明確な要望が出た。利用者1名の SPA に対して、レイヤーのボイラープレット(ADR-0002 自身が「影響」欄で認めていたトレードオフ)は利益より重い。

## 決定(何をどうするか)

クリーンアーキテクチャの**エッセンスを2つのルールに凝縮**し、ディレクトリは役割ベースのフラットな構成にする。

```
src/
  core/        # ゲームのルール全部(純粋な TypeScript。ここが旧 domain+usecase)
    curriculum/   # Stage, Lesson, 解放 (R5-R7)
    practice/     # Exercise, Attempt, Medal 判定 (R1-R4)
    generation/   # お題の自動生成 (R6, R13-R14)
    progression/  # Streak, フリーズ, XP, 実績 (R8-R18)
    analytics/    # 弱点判定 (R19)
    ports.ts      # 外界とのインターフェース: VimEngine, ProgressStore, Clock, RandomSource
  vim/         # VimEngine の実装(エミュレーションライブラリのラップ)
  storage/     # ProgressStore の実装(IndexedDB、エクスポート/インポート)
  ui/          # 画面・コンポーネント・演出・ルーティング
  main.ts(x)   # 組み立て(core と実装を配線)と起動
```

**守るルールは2つだけ:**

1. **core の純粋性**: `core/` は `core/` 以外を import しない。フレームワーク・ブラウザ API(`Date.now()`・`Math.random()` 含む)禁止。外界が必要なものは `ports.ts` のインターフェース経由。
2. **エンジンはポートの向こう**: ui は vim ライブラリの API を直接呼ばず、`VimEngine` インターフェースを通す(ADR-0003 は引き続き有効)。

それ以外(ui が storage を直接使う、DTO 変換層を挟まない等)は自由。迷ったら「シンプルに書けるほう」を選ぶ。

## 検討した代替案(採用しなかった案と理由)

- **ADR-0002 の4層構成を維持**: 依存ルールの厳密さは最高だが、オーナーの開発体験(1機能=1〜2ディレクトリで完結してほしい)を損なう。個人プロジェクトでは「続けて開発できること」自体が最重要要件。
- **構造ルールなし(components/ utils/ 等)**: ルール1(core の純粋性)まで捨てると、業務ルール R1〜R19 のテストが UI 依存になり、vim エンジン差し替えの保険(ADR-0003)も失う。ここだけは譲らない。

## 影響(この決定で生じるトレードオフ)

- (+) 1機能の変更がほぼ `core/<context>/` + `ui/` の2箇所で完結する。認知負荷が低い。
- (+) 守るべきルールが2つに減り、lint 検査も「core は core 以外を import しない」の1本で済む。
- (+) core の純粋性は維持されるため、業務ルールのテスト容易性・エンジン差し替え可能性は失わない。
- (−) ui と storage の間に変換層が無いため、保存形式の変更が ui に波及しうる(ProgressStore の型を core に置くことで緩和)。
- (−) CLAUDE.md の開発原則(4層の明示)とは字義通りには一致しない。本 ADR を「このプロジェクトにおけるクリーンアーキテクチャの適用形」とする(CLAUDE.md に注記済み)。
