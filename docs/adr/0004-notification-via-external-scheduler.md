# 0004: 通知はアプリ外のスケジューラ + ntfy.sh で実現

- ステータス: 採用
- 日付: 2026-07-11

## コンテキスト(なぜ判断が必要か)

要件として「サボり防止のスマホ通知」があり、方式はプロダクトオーナーが ntfy.sh 等の外部通知サービス案を選択済み(docs/project-overview.md)。一方でアプリはバックエンドレス(ADR-0001)であり、ブラウザを閉じた状態のアプリ自身は定時に何もできない。誰が毎日20時(暫定 P7)に通知を発火するかを決める必要がある。

## 決定(何をどうするか)

- 通知経路: スケジューラ → ntfy.sh へ HTTPS で publish → スマホの ntfy アプリが購読・表示。
- スケジューラは GitHub Actions の cron ワークフローとする(このリポジトリに置く。無料枠で十分)。
- 通知は固定文言のリマインドのみ。進捗・成績データを一切含めない(domain.md 禁止事項)。アプリ本体と通知系はデータを共有しない完全分離とする。
- 「今日やったかどうか」を通知側は知らない(知るには進捗の外部送信が必要になり禁止事項に反する)。やった日にも通知は来るが、リマインドとして許容する。
- ntfy のトピック名は推測されにくいランダム文字列とし、リポジトリを公開する場合は GitHub Secrets に置く。

## 検討した代替案(採用しなかった案と理由)

- **PWA + Web Push**: アプリ内から「今日は完了」を反映した賢い通知が可能だが、push 送信サーバー(または有料サービス)と購読管理が必要で、バックエンドレス方針に反する。iOS の Web Push はホーム画面追加が前提で信頼性も劣る。
- **スマホ標準のリマインダー/アラーム**: 実装ゼロで済むが、アプリのアイコン・文言で「vim-dojo からの呼び出し」感を出せず、飽き防止の演出余地がない。ユーザーが手動で無効化しても検知できない点は ntfy でも同じ。
- **cron-job.org 等の外部 cron サービス**: GitHub Actions と同等だが、リポジトリ外に管理対象が増える。コードと同じ場所で管理できる Actions を優先。

## 影響(この決定で生じるトレードオフ)

- (+) アプリ本体はローカル完結のまま、実装は YAML 1ファイル + ntfy アプリの購読設定のみ。
- (+) 通知文言・時刻の変更が Git で管理される。
- (−) 「今日は既にやった」日にも通知が来る(進捗非送信の代償として受容)。
- (−) GitHub Actions の cron は数分〜数十分の遅延がありうる(リマインド用途では許容)。
- (−) リポジトリが GitHub にあることが前提になる(現在ローカルのみ。インフラ構築フェーズで GitHub リポジトリ化する)。
