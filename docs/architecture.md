# アーキテクチャ設計 — vim-dojo

最終更新: 2026-07-11

前提: `docs/project-overview.md`(ローカル完結・サクサク動作・設計を固めてから実装)と `docs/domain.md`(境界づけられたコンテキスト5つ)。既存コードは無いため、本ドキュメントは現状の観察ではなく設計方針そのものである。

## 採用アーキテクチャ

**バックエンドレス SPA + core/shell の簡易構成。**

- アプリはブラウザ内で完結する静的配信の SPA。サーバー側のアプリケーションコードを持たない(→ ADR-0001)。
- クリーンアーキテクチャは**エッセンスのみ**を採用する: ゲームのルール(R1〜R19)を純粋な `core/` に隔離し、UI・vim エンジン・ストレージ(shell)から一方向に依存する。フル4層のディレクトリ分割はしない(→ ADR-0005。ADR-0002 を置換)。
- vim エミュレーションエンジンは `VimEngine` ポート経由で利用し、差し替え可能に保つ(→ ADR-0003)。
- スマホ通知はアプリ本体から完全に分離した外部構成(スケジューラ + ntfy.sh)で実現する(→ ADR-0004)。

採用理由: 利用者1名・ローカル保存・起動の速さ最優先という要件に対し、サーバーは純粋なコストにしかならない。ドメインルールは豊富にある一方、開発者は1名であり「1機能の変更が少数のファイルで完結するわかりやすさ」が継続開発の前提条件になる。守る価値が最も高い分離(ルールの純粋性・エンジンのポート化)だけを残した。

## ディレクトリ構成

役割ベースのフラットな構成(ADR-0005)。

```
src/
  core/        # ゲームのルール全部。純粋な TypeScript(フレームワーク・ブラウザ API 禁止)
    curriculum/   # Stage, Lesson, 解放 (R5-R7)
    practice/     # Exercise, Attempt, Medal 判定 (R1-R4)
    generation/   # お題の自動生成 (R6, R13-R14)
    progression/  # Streak, フリーズ, XP, 実績 (R8-R18)
    analytics/    # 弱点判定 (R19)
    ports.ts      # 外界とのインターフェース: VimEngine, ProgressStore, Clock, RandomSource
  vim/         # VimEngine の実装(エミュレーションライブラリのラップ)
  storage/     # ProgressStore の実装(IndexedDB、エクスポート/インポート)
  ui/          # 画面・コンポーネント・演出・ルーティング
  main.ts(x)   # 組み立て(core と実装を配線)と起動
```

| 場所          | 責務                                                             | 置いてはいけないもの                                                         |
| ------------- | ---------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| core          | 用語集の概念と業務ルール R1〜R19、ユースケースの手順、ポート定義 | フレームワーク・ブラウザ API への import(`Date.now()`・`Math.random()` 含む) |
| vim / storage | `ports.ts` のインターフェースの実装                              | 業務ルール(判定・計算)そのもの                                               |
| ui            | 画面・演出・入力の受け付け。core の関数を呼ぶ                    | 業務ルールの再実装(メダル判定を UI 側で計算し直す等)                         |

クリーンアーキテクチャ4層との対応(開発原則の適用形): core = domain + usecase、vim / storage / ui = adapter + infra。

## 依存方向

**守るルールは2つだけ:**

1. **core の純粋性**: `core/` は `core/` 以外を import しない。外界(時刻・乱数・エンジン・保存)が必要なものはすべて `ports.ts` のインターフェース経由で受け取る。
2. **エンジンはポートの向こう**: ui は vim ライブラリの API を直接呼ばず、`VimEngine` インターフェースを通す。

```
ui ──▶ core ◀── vim, storage(ports.ts の実装)
main ──▶ 全部(配線のみ)
```

それ以外(ui が storage を直接使う等)は自由。迷ったら「シンプルに書けるほう」を選ぶ。

**違反の検出方法:**

- lint で「`core/` 内から `core/` 外への import 禁止」を機械的に検査する(1ルールで済む。具体ツールは技術選定・CI 構築フェーズで確定)。CI で必須チェックにする。
- レビュー観点: 業務ルールの計算・判定が ui / vim / storage に書かれていないか。

## モジュール分割

`docs/domain.md` の境界づけられたコンテキストをそのまま採用する:

| モジュール       | 対応コンテキスト | 境界の根拠                                                                       |
| ---------------- | ---------------- | -------------------------------------------------------------------------------- |
| core/curriculum  | カリキュラム     | 変更理由が「学習内容の設計」。コンテンツ追加で頻繁に触る                         |
| core/practice    | 演習             | 変更理由が「計測・判定ルール」。vim エンジン差し替えの影響を遮断する境界         |
| core/generation  | 出題生成         | 変更理由が「生成アルゴリズム」。シード決定性という独自の不変条件を持つ           |
| core/progression | 進捗・動機づけ   | 変更理由が「ゲーミフィケーション調整」(XP量・フリーズ等の暫定判断 P1〜P3 が集中) |
| core/analytics   | 分析             | 変更理由が「弱点判定・可視化の改良」。他モジュールの読み取り専用消費者           |

コンテキストをまたぐ処理(例: お題クリア → XP 付与 → 弱点記録)は、core 内の呼び出しで素直に合成してよい。厳密なコンテキスト間 import 禁止は課さない(シンプルさ優先)。

## 外部サービス連携

| 連携先        | プロトコル                      | 方向                              | 障害時の扱い                                                                     |
| ------------- | ------------------------------- | --------------------------------- | -------------------------------------------------------------------------------- |
| ntfy.sh(通知) | HTTPS(スケジューラから publish) | アプリ本体とは無関係(外部→スマホ) | ベストエフォート。未達でもアプリ・ストリーク判定に影響なし(domain.md 例外ケース) |

アプリ本体からの外部通信は**ゼロ**(静的アセットの取得を除く)。進捗・成績データを外部に送信しない(domain.md 禁止事項)。

## ADR 作成ルール

- **いつ書くか**: アーキテクチャスタイル、レイヤー/依存ルールの変更、外部サービスの追加、データ永続化方式の変更など「後から覆すのが高くつく判断」をする前に書く。
- **置き場所と番号**: `docs/adr/NNNN-<slug>.md`。番号は 0001 から連番(欠番を再利用しない)。
- **テンプレート**: `skills/architecture-design/SKILL.md` 記載のものを使う(ステータス / コンテキスト / 決定 / 代替案 / 影響)。
- **ステータス遷移**: 提案 → 採用 → 廃止(廃止時は後継 ADR へのリンクを残す)。
