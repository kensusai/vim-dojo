# プロジェクト概要 — vim-dojo

最終更新: 2026-07-11(プロダクトオーナーへのヒアリングに基づく)

このドキュメントでは **[事実]**(オーナーに確認済み)と **[仮説]**(未確認の想定)を区別して記述する。矛盾する実装依頼を受けた場合は、着手前に矛盾を指摘すること。

## 何を作るのか

**一言説明**: vim が苦手な人が、楽しく・飽きずに・継続的に vim 操作を鍛えられるブラウザゲーム。 **[事実]**

**詳しい説明**: レッスンで新しい vim コマンドを学び、ドリルで反復し、毎日のデイリーチャレンジで定着させる**ハイブリッド型**の学習ゲーム。ブラウザ内の vim 演習環境でお題(テキスト変換課題など)をこなし、キーストローク効率でスコアリングされる。学習の体系性(vimtutor 的なカリキュラム)とゲームの反復性(VimGolf 的なスコアアタック)を組み合わせる。 **[事実]**

### 学習ループ(コア設計)

```
レッスン(新コマンド解説 + ブラウザ内演習)
      │ クリアでコマンド解放
      ▼
ドリル(解放済みコマンドのお題を反復、スコア↑)
      │
      ▼
デイリーチャレンジ(毎日1問・ストリーク)
      │
      ▼
弱点分析 → 苦手コマンドがドリルに再出題
```

## 誰のためのものか

- **主ユーザー**: オーナー自身(vim/Neovim 初学者〜中級手前。macOS / Neovim / tmux 環境の開発者)。まず自分専用として作る。 **[事実]**
- 将来的な公開・他ユーザー提供は視野に入れるが、初期スコープ外。 **[事実]**

## 解決する課題

- vim の習得は挫折しやすい: vimtutor は一度きりで反復動機がなく、実務でいきなり使うとストレスで元のエディタに戻ってしまう。 **[事実(オーナーの実感)]**
- 既存サービスの不足を埋める: VimGolf は初心者に厳しく(知らないコマンドは自力で調べる必要がある)、レッスン型は修了後のリプレイ動機が弱い。両者の組み合わせで「学ぶ→反復する→毎日続ける」を1つのループにする。 **[仮説]**

## 主要機能(優先順位付き)

### P1 — コア(これが無いと成立しない)

1. **ブラウザ内 vim 演習環境**: normal / insert / visual モード、対象スキル範囲のコマンドが動作するエディタ画面。キー入力の記録とキーストローク数の計測。
2. **レッスン(カリキュラム)**: 以下の4段階を体系的にカバーする。クリアで次のコマンドが解放される。 **[事実]**
   - 基本移動・モード操作(hjkl, w/b/e, 0/$, gg/G, f/t など)
   - 編集オペレータ + テキストオブジェクト(d/c/y × iw, i", ip など)
   - 検索・置換・グローバルコマンド(/, ?, :s, :g, 正規表現)
   - 上級(レジスタ、複数ファイル/バッファ・ウィンドウ操作など実務的な操作系。※マクロはいったん対象外 **[事実]**)
   - **素の vim の文法をメイン**とする。挙動の基準は Neovim のモダンな標準(例: `Y` は `y$` 相当)。レッスン解説には豆知識としてモダン環境(LazyVim 等)との差異注記(例: LazyVim では `s` は flash.nvim のジャンプ)を入れられるようにする(任意)。 **[事実]**
3. **ドリル**: 解放済みコマンドを使うお題の反復。キーストローク数によるスコアリング(パー/最適解との比較)。

### P2 — 継続の仕掛け(飽き対策。本プロダクトの存在意義)

4. **デイリーチャレンジ + ストリーク**: 毎日1問の日替わりお題と連続日数の記録。お題はシードによる自動生成とし、まったく同じ課題が繰り返されないようにする。ストリークは厳格リセットではなく猶予(フリーズ)あり。 **[事実]**
   - サボり防止のスマホ通知は、ntfy.sh 等の外部通知サービスへの定時リマインドで実現する方針(アプリ本体はローカル完結を維持し、push 送信サーバーを自前で持たない)。 **[事実]**
5. **レベル・XP・実績バッジ**: 経験値によるレベルアップ、条件達成でバッジ解除。 **[事実]**
6. **ローカルデータ保存**: 進捗・スコア・ストリークをブラウザローカルに永続化(バックエンド・アカウント不要)。 **[事実]**

### P3 — 分析・最適化

7. **成長の可視化**: キーストローク効率の推移グラフ。
8. **弱点分析**: 苦手コマンドの自動検出と、ドリルへの再出題。

### やらないこと(初期スコープ外)

- アカウント・認証・サーバー側データ保存 **[事実]**
- ランキング・対戦など他ユーザーとの比較機能 **[事実]**
- マクロ(q による記録・再生)の学習(いったん対象外。将来カリキュラムに追加する可能性あり) **[事実]**
- モバイルでのフル演習(キーボード前提のため、演習はデスクトップブラウザ専用) **[事実]**
  - ただしオーナーには「スマホでも学習できると理想」という要望がある。キーボード不要の軽い学習モード(コマンドクイズ等)を将来のスマホ向け機能として検討する余地あり。 **[仮説]**
- vimscript / Lua 設定・プラグインの学習(操作スキルに集中) **[仮説]**
- LazyVim プラグイン操作(flash のジャンプ、surround、Space リーダーのメニュー等)のエミュレータでの再現(レッスン解説での注記に留める。エンジン拡張のコストに見合わない) **[事実]**

## 非機能要件

- **UI/UX**: モダンでわくわくするデザインであること。 **[事実(要望)]** 具体化の方向性(いずれも仮説、デザインモック作成時に確定):
  - ゲームらしい高揚感のあるマイクロインタラクション — メダル獲得・レベルアップ・ストリーク更新時の演出、XP バーのアニメーションなど「続けたくなる」瞬間を演出で強化する。
  - vim / ターミナルの世界観に合うモダンなビジュアル(ダークテーマ基調、等幅フォントのエディタ面と、リッチな UI 面のコントラスト)。
  - 演出はキー入力を絶対にブロックしない(下記レイテンシ要件が常に優先)。派手さより「反応の速さ+気持ちいいフィードバック」を優先する。
- **キー入力レイテンシ**: エディタ演習はタイピング体験そのものなので、キー入力への反応は体感即時(目安 16ms 以内 / 1フレーム)であること。 **[仮説]**
- **起動の速さ**: ローディングが長いと「毎日開く」習慣が途切れるため、アプリを開いてから演習開始までがサクサクであること。 **[事実(要望)]** 目安: 再訪時はキャッシュにより即表示、初回ロードも数秒以内。デイリーチャレンジには最小の操作数で到達できること。数値目標は技術選定後に計測とセットで確定する。 **[仮説]**
- **オフライン/ローカル完結**: バックエンドなしで動作し、データはブラウザローカルに保存。ブラウザのデータ消去で進捗が消えるリスクはエクスポート/インポートで緩和する。 **[仮説]**
- **キーボード配列**: US 配列前提。JIS 配列対応は不要。 **[事実]**
- **スケール想定**: 利用者1名(オーナー)。可用性・負荷対策は不要。 **[事実]**
- **セキュリティ**: 個人情報・秘密情報を扱わない。公開リポジトリに個人名・秘密情報を含めない(グローバル方針)。 **[事実]**

## 成功条件

以下の3つを成功指標とする。 **[事実]**

1. **習慣化**: ストリークが続くこと(計測: 連続日数。マイルストーン例: 30日連続)。
2. **実務での vim 上達**: 実際の編集作業がマウス・矢印キーなしでこなせるようになること(アプリ内の代理指標: 同一お題のキーストローク数の改善推移)。
3. **カリキュラム修了**: 基本移動〜上級まで全レッスンをクリアすること。

## 制約条件

- **進め方**: 設計(カリキュラム全体・XP・実績・弱点分析まで)を固めてから一気に実装する方針。開発ワークフロー(ドメイン理解→アーキテクチャ設計→技術選定→…)に沿って設計フェーズを先に完了させる。 **[事実]**
- **期限・予算**: 個人プロジェクトのため厳密な期限なし。ランニングコストはローカル完結のためほぼゼロ想定。 **[事実]**
- **技術**: 選定済み(`docs/tech-stack.md` / ADR-0006, 0007)。TypeScript + Vite + React + CodeMirror 6(vim 拡張)ほか。オーナー環境は macOS / mise(node, python, java)。 **[事実]**
- **本プロジェクトの開発原則**: クリーンアーキテクチャ / ドキュメント重視 / パフォーマンス重視 / 継続的リファクタリング / 型安全重視(CLAUDE.md 参照)。 **[事実]**

## 未確定事項 — プロダクトオーナーへの質問リスト

次フェーズ(ドメイン理解・アーキテクチャ設計・技術選定)までに確認したい事項:

1. **デイリーチャレンジの自動生成の設計**: シード自動生成は決定済み。生成アルゴリズム(テキスト素材・変換パターンの組み合わせ方、難易度制御、解放済みコマンドとの整合)の具体化。→ DB設計(データモデル)〜実装フェーズで詰める。
2. **UI 言語**: 日本語のみで暫定確定(domain.md P8)。将来公開する場合の英語対応は必要になってから。

### 確認済み(2026-07-11 回答)

- マクロはいったんカリキュラム対象外。レジスタは一度対象外としたが、最終的にカリキュラムに含めることにした。上級は「レジスタ+複数ファイル/バッファ・ウィンドウ操作」の実務的な操作系。
- デイリーチャレンジはシードによる自動生成(同一課題の繰り返しを避ける)。
- キーボード配列は US 前提で OK。
- ストリークは猶予あり。ただしサボり防止のスマホ通知を行いたい。スマホでのフル演習はキーボードがないため見送り(軽量学習モードは将来検討)。
- スマホ通知は ntfy.sh 等の外部通知サービス方式(案b)を採用。アプリ本体はローカル完結を維持。
- 起動・表示のサクサク感を重視(ローディングが長いと続かないため)。非機能要件に追加済み。
- モダンでわくわくする UI/UX を重視する。非機能要件に追加済み(具体化はデザインモック作成時)。
- vim エンジンは CodeMirror 6 + `@replit/codemirror-vim` を採用(ADR-0006)。UI フレームワークは React(オーナー選択、ADR-0007)。
- 素の vim の文法をメインとする(オーナーは LazyVim 未使用、興味があるのみ)。挙動基準は Neovim 標準、LazyVim 等との差異は豆知識注記に留める(プラグイン操作の再現はしない)。
