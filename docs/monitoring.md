# 監視設計 — vim-dojo

最終更新: 2026-07-11(M8)

利用者1名・ローカル完結・テレメトリなし(docs/tech-stack.md)のため、サーバー監視は存在しない。「監視」に相当するのは以下の3点。

## 1. 進捗データの破損検知

- IndexedDB からの読み込みは毎回 zod でパースし、壊れたデータは**起動時に大きな音で失敗**する(docs/database.md)。App はエラー画面で「エクスポート JSON からのインポートで復旧」を案内する。
- 復旧手段: ホーム画面フッターのエクスポート/インポート(UC6)。エクスポート促しとして最終エクスポート日時を常時表示。

## 2. CI / デプロイの失敗検知

- push ごとの CI(lint→format→型→テスト→ビルド)と Deploy の結果は GitHub の通知で把握する。落ちたら直すまで先に進まない(docs/ci.md 失敗時の運用)。
- 通知 cron(notify.yml)の失敗も Actions の実行履歴で確認できる。通知はベストエフォート(domain.md 例外ケース)なので、失敗してもアプリ・ストリーク判定には影響しない。

## 3. クライアント側の異常

- テレメトリを送らないため、実行時エラーはオーナー自身の DevTools で確認する(利用者=開発者なので成立する)。
- 定期的な健全性確認は e2e ドライブ(`e2e/drive-m*.mjs`)をローカルで実行する。ユニットテストが通っていても実ブラウザだけで起きる不具合があるため(M5・M7 で実証済み)、CI が push ごとに E2E スモーク(`e2e/smoke.spec.ts`)を自動実行する。より深いドライブ(`e2e/drive-m6/m7.mjs`・`verify-live.mjs`)は**大きめの変更のリリース前に手動で流す**ことを運用ルールとする。

## 障害対応手順

| 症状                           | 対応                                                                                                 |
| ------------------------------ | ---------------------------------------------------------------------------------------------------- |
| アプリが起動しない(エラー画面) | エクスポート済み JSON をインポート。無ければ DevTools で IndexedDB `vim-dojo` を削除(進捗は失われる) |
| デプロイが失敗                 | Actions のログ確認 → 修正 push。サイトは前回成功時の内容のまま                                       |
| 通知が来ない                   | Actions の notify 実行履歴を確認 → Secret `NTFY_TOPIC` の設定と ntfy アプリの購読を確認              |
