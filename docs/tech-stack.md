# 技術選定 — vim-dojo

最終更新: 2026-07-11

前提: ADR-0001(バックエンドレス SPA)/ ADR-0005(core/shell 簡易構成)/ 非機能要件「起動サクサク・キー入力即応・モダンでわくわくする UI」。

**運用ルール**: 新しい依存を追加する前にこのドキュメントと照合する。方針にないものは追加前に提案として挙げる(理由を要件・制約に紐づけること)。バージョンの正は `package.json` と mise 設定であり、本ドキュメントには固定バージョンを書かない。

## 言語 / ランタイム

- **採用**: TypeScript(`strict: true`)。Node.js は mise で管理(グローバル設定に準拠。brew の node を直接参照しない)。
- **理由**: 開発原則「型安全重視」(判別可能ユニオン・網羅性チェック・branded type)を満たす実質唯一の選択。R1〜R19 のルールを型で守る。
- **不採用**: JavaScript(型安全原則に反する)。
- **見直し条件**: なし(前提)。

## ビルド / UI フレームワーク

- **採用**: Vite + React。
- **理由**: React はオーナーが選択(情報量・エコシステム・AI エージェントの習熟度が最大)。Vite は起動サクサク要件に必要なコード分割・軽量ビルドが素直にできる。
- **不採用**: Svelte / Vue(バンドルは軽いが情報量で劣り、オーナーの選好も React)。Next.js 等のメタフレームワーク(サーバー機能が不要。静的 SPA に SSR/ルーティング基盤は過剰)。Monaco ベースの構成(下記 vim エンジン参照)。
- **見直し条件**: 初回ロードが目標(数秒以内)を超え、コード分割でも解決しない場合にバンドル軽量化を再検討。

## vim エンジン(→ 詳細は ADR-0006)

- **採用**: CodeMirror 6 + `@replit/codemirror-vim`。`src/vim/` で `VimEngine` ポート(ADR-0003)にラップし、ライブラリ API を ui/core に漏らさない。
- **理由**: カリキュラム範囲(移動・オペレータ・テキストオブジェクト・検索置換・レジスタ)をカバーする実績あるエミュレーション。CodeMirror 6 本体が軽量・モジュール式で起動サクサク要件と合う。
- **不採用**: monaco-vim(Monaco が重い。VS Code 由来の巨大バンドルは即起動に不利)/ vim.wasm(再現度は完璧だが WASM ロードが重く、キーストローク計測・バッファ検査の統合が困難)/ 自前エンジン(工数が支配的。ポート化してあるので将来の選択肢として温存)。
- **注意**: ステージ4「複数ファイル/バッファ操作」はエンジンの守備範囲外として、アプリ側(ui + core)でバッファ切替を実装する。エンジンへの要求は常に単一バッファ。
- **見直し条件**: レッスン実装中に必須コマンドの挙動差異が見つかったら、まず拡張ポイントで補完し、根本的に足りない場合に自前エンジン(ADR-0003 の差し替え)を再検討。

## スタイリング / アニメーション

- **採用**: Tailwind CSS + Motion。
- **理由**: 「モダンでわくわくする UI」を1人で速く作るための構成。Tailwind はダークテーマ基調のデザインシステムを CSS 変数と併用で低コストに維持できる。Motion はメダル獲得・レベルアップ等のマイクロインタラクション(非機能要件)の実装コストが最小。
- **不採用**: CSS Modules / vanilla-extract(自由度は高いが1人開発では速度で劣る)/ UI コンポーネント集(MUI 等: 見た目が「管理画面」になりがちで、わくわく要件・バンドル軽量と相性が悪い)。
- **見直し条件**: 演出がキー入力レイテンシを阻害したら演出側を削る(非機能要件の優先順位に従う)。

## 状態管理 / ストレージ

- **採用**: zustand(画面をまたぐ Progress の共有)+ IndexedDB(`idb` ラッパー)。
- **理由**: zustand はボイラープレートほぼゼロで「シンプルな実装」方針に合う。進捗は試行履歴が増え続けるため localStorage(文字列一発保存)より IndexedDB。`idb` は Promise 化だけの薄いラッパーで、core の `ProgressStore` ポート実装が数十行で書ける。
- **不採用**: Redux(過剰)/ React Context のみ(再レンダー制御と購読の記述量で zustand に劣る)/ Dexie(高機能だが、クエリは core 側でメモリ上で行うため不要)/ localStorage(容量・構造化で不利。ただしエクスポートは JSON ファイルで行う)。
- **見直し条件**: 試行履歴の読み込みが起動を遅くしたら、履歴の遅延ロード・集計キャッシュを導入。

## スキーマ検証(ストレージ境界)

- **採用**: zod。保存データ(IndexedDB・エクスポート JSON)の形は zod スキーマを唯一の定義とし、型は `z.infer` で生成、ID は `.brand()` で branded type にする(`docs/database.md` 参照)。
- **理由**: 型安全原則「外部データは境界で一度だけパースする」の実装手段。IndexedDB のデータは旧バージョン・破損がありうる外部入力として扱う。
- **不採用**: 手書きの検証関数(スキーマと型の二重管理になる)/ valibot(軽いが情報量で zod。バンドルに効くのは境界1箇所のみ)。
- **見直し条件**: バンドルサイズ計測で zod が支配的になったら valibot / zod mini を再検討。

## 認証・認可

- **対象外**: 利用者1名・ローカル完結(ADR-0001)。公開判断が出た時点で ADR を起こす。

## インフラ / デプロイ

- **採用**: GitHub Pages(静的配信)+ GitHub Actions(ビルド・デプロイ・通知 cron ※ADR-0004)。
- **理由**: リポジトリと同じ場所で完結し無料。通知スケジューラ(ADR-0004)が GitHub Actions 前提のため、リポジトリの GitHub 化と合わせて一石二鳥。
- **不採用**: Cloudflare Pages / Netlify(いずれも良いが管理対象が増える。GitHub Pages で不足が出たら再検討)。
- **見直し条件**: プライベートリポジトリのまま Pages を使う場合の制約(プラン)に当たったら Cloudflare Pages を再検討。
- **補足**: vite-plugin-pwa による Service Worker キャッシュで再訪時の即時起動(非機能要件)を実現する。導入は CI 構築フェーズの後(キャッシュはデバッグを難しくするため、開発初期は入れない)。

## テスト

- **採用(ユニット)**: Vitest + @testing-library/react。core の業務ルール R1〜R19 は UI なしの純粋関数テストとして網羅する(core/shell 構成の主目的)。
- **未選定(E2E)**: 候補は Playwright。選定基準: 「レッスン開始→キー入力→クリア判定→保存」の一連が実ブラウザで検証できること。導入タイミングは最初の画面が動いてから(CI 構築フェーズで再判断)。
- **不採用**: Jest(Vite との統合で Vitest に一日の長)。
- **補助**: fake-indexeddb(dev)— storage 層の結合テスト(保存→リロード→復元)を Node 上で実行するため。ブラウザ実物での検証は E2E 導入時に統合。

## lint / format

- **採用**: ESLint + Prettier。ESLint の import 制約ルールで「`core/` は `core/` 以外を import しない」(ADR-0005 の唯一の必須境界)を機械検査する。
- **不採用**: Biome(高速だが import 境界ルールの表現力で ESLint が必要になり、二重管理を避ける)/ dependency-cruiser(ルール1本のために別ツールを増やさない)。
- **見直し条件**: lint が遅くて開発リズムを崩したら Biome + 最小 ESLint の併用を再検討。

## 監視 / ログ基盤

- **対象外**: 個人利用・ローカル完結・外部送信禁止(domain.md)のため、テレメトリは導入しない。デバッグはブラウザ DevTools で足りる。監視設計フェーズでは「進捗データの破損検知(ロード時の parse 検証)」のみ扱う。
