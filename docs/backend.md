# core 実装ルール(バックエンド開発フェーズの適用形)— vim-dojo

最終更新: 2026-07-11

このプロジェクトにはサーバーが無い(ADR-0001)。開発ワークフローの「バックエンド開発」フェーズは **`src/core/` の実装**として適用する。core はゲームのルール全部(R1〜R19)を持つ純粋 TypeScript であり、サーバーで動くビジネスロジックの完全な代替物である。

## 採用アーキテクチャ / ディレクトリ

- ADR-0005 の core/shell 構成。`src/core/<コンテキスト>/`(curriculum / practice / generation / progression / analytics)+ `src/core/ports.ts` + `src/core/ids.ts`。
- 新規ファイルの置き場所: 業務ルールの実装は必ず該当コンテキストへ。どのコンテキストにも属さない共有型は乱発しない(現状は `ids.ts` のみ)。

## API 設計(= core の公開関数の規約)

- HTTP API は無い。「API」= core が export する関数・型。
- 公開関数は**純粋関数**または**ポート注入のファクトリ**(例: `startPracticeSession({ exercise, engine, clock })`)。暗黙の外界アクセス(`Date.now()` 等)は禁止 — ESLint が機械検査する。
- 引数・戻り値はプレーンなデータ(interface / union)。クラス階層・フレームワークオブジェクトを公開しない。
- ID は branded type(`ids.ts`)で受け渡す。裸の string で境界を越えない。
- doc コメントに目的・対応するルール番号(R1〜R19)・失敗モードを書く。

## エラーハンドリング

- **プログラミングエラー / コンテンツのバグ**(不変条件違反: par が非正、initial === target 等)→ `RangeError` / `Error` を throw。呼び出し側のバグなので回復しない。ガードは境界の関数(`assertValidExercise` 等)に置く。
- **業務上ありうる状態**(お題の放棄、フリーズ切れによるストリークリセット等)→ 例外ではなく**戻り値の union / null** でモデル化する。「不正な状態を表現不能に」が第一、次に判別可能ユニオン。
- エラーを握りつぶさない。catch して無視するコードを core に書かない。

## バリデーション(どの層で何を検証するか)

- **storage 境界**: IndexedDB / インポート JSON は zod で一度だけパース(docs/database.md)。core には型付きの値だけが入る。
- **core**: 型で表現できない不変条件(数値範囲・組み合わせ)を assert 関数で守る。
- **ui**: 検証しない。core の判定結果を表示するだけ。

## 認証・認可 / バックグラウンドジョブ / ログ

- 対象外。利用者1名・ローカル完結(ADR-0001)のため認証・認可は存在しない。ジョブは通知 cron のみでアプリ外(ADR-0004)。core に console 出力を書かない(テスト容易性と純粋性のため)。

## パフォーマンス

- 応答時間の目標: core の判定・計算はすべて**キー入力1回のフレーム内(16ms)で完了**すること(非機能要件)。現状の関数はバッファ長 O(n) の比較・O(1) の算術のみで、余裕がある。
- クリア判定(R1)はキーストロークごとに走るホットパス。バッファ全文比較 O(n) で十分速いが、お題のバッファは数百文字までを想定(コンテンツ設計の前提として明記)。長大バッファのお題を作らない。
- attempts の集計(分析)は全件をメモリで処理してよいが、**起動パスでは実行しない**(docs/database.md のパフォーマンス方針)。

## テスト方針

- core の業務ルール(R1〜R19)は**ルール番号をテスト名・コメントに明記**した純粋関数テストで網羅する。
- 外部依存はポートのフェイクで注入する: `FakeVimEngine`(テスト内定義)、`fakeClock`。時刻・乱数の実物をテストで使わない(フレーキー防止)。
- 異常系(不変条件違反・放棄・二重操作)のテストを各公開関数に最低1つ含める。
- フェイクが3箇所で重複したら共有ヘルパーに昇格させる(三度目の法則)。
